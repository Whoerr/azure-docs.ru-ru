# <a name="azure-security-baseline-for-automation"></a>Базовый уровень безопасности Azure для автоматизации

Этот базовый план безопасности применяет рекомендации из [тестового показателя безопасности Azure версии 1,0](../security/benchmarks/overview-v1.md) к службе автоматизации Azure. Azure Security Benchmark содержит рекомендации по обеспечению безопасности облачных решений в Azure.
Содержимое группируются по **элементам управления безопасностью** , определенным в ходе тестирования системы безопасности Azure, и связанным рекомендациям, применимым к службе автоматизации Azure. **Элементы управления** , неприменимые к службе автоматизации Azure, были исключены.

 
Чтобы узнать, как служба автоматизации Azure полностью сопоставляется с тестовым показателем безопасности Azure, ознакомьтесь с [полным файлом сопоставления базовых показателей безопасности службы автоматизации Azure](https://github.com/MicrosoftDocs/SecurityBenchmarks/tree/master/Azure%20Offer%20Security%20Baselines).

### <a name="11-protect-azure-resources-within-virtual-networks"></a>1,1: защита ресурсов Azure в виртуальных сетях

**Руководство**. учетная запись службы автоматизации Azure еще не поддерживает частную ссылку Azure для ограниченного доступа к службе через частные конечные точки. Модули Runbook, которые проходят проверку подлинности и запускаются для ресурсов в Azure, выполняются в песочнице Azure и используют общие серверные ресурсы, которые корпорация Майкрософт несет за изоляцию друг от друга. их сеть не ограничена и имеет доступ к общедоступным ресурсам. В настоящее время служба автоматизации Azure не поддерживает интеграцию виртуальной сети для частных сетей, помимо поддержки гибридных рабочих ролей Runbook. Этот элемент управления неприменим, если вы используете встроенную службу без гибридных рабочих ролей Runbook.

Чтобы расширить возможности изоляции модулей Runbook, можно использовать гибридные рабочие роли Runbook, выполняющиеся на виртуальных машинах Azure. При создании виртуальной машины Azure необходимо создать виртуальную сеть (VNet) или использовать существующую, а также настроить виртуальные машины с подсетью. Убедитесь, что все развернутые подсети имеют группу безопасности сети, к которой относятся элементы управления доступом к сети, относящиеся к доверенным портам и источникам приложений. Требования к конкретной службе см. в статье рекомендации по безопасности для конкретной службы. Кроме того, при наличии определенного требования можно использовать брандмауэр Azure для его удовлетворения.

- [Виртуальные сети и виртуальные машины в Azure](../virtual-machines/network-overview.md)

Если вы используете гибридные рабочие роли Runbook, поддерживающие виртуальные машины Azure, убедитесь, что подсеть, содержащая эти рабочие роли, включена с группой безопасности сети (NSG), и настройте журналы потоков для пересылки журналов в учетную запись хранения для аудита трафика. Некоторые преимущества Аналитики трафика — это возможность визуализировать сетевые активности и определять горячие участки, выявлять угрозы безопасности, анализировать шаблоны потоков трафика и выявлять некорректные сетевые настройки.

Хотя правила NSG и определяемые пользователем маршруты не применяются к частной конечной точке, журналы потоков NSG и данные мониторинга для исходящих соединений по-прежнему поддерживаются и могут использоваться.

### <a name="14-deny-communications-with-known-malicious-ip-addresses"></a>1,4: запретите обмен данными с известными вредоносными IP-адресами

**Руководство**. в настоящее время служба автоматизации Azure не поддерживает интеграцию виртуальной сети для частных сетей, помимо поддержки гибридных рабочих ролей Runbook. Этот элемент управления неприменим, если вы используете встроенную службу без гибридных рабочих ролей Runbook.

Если вы используете гибридные рабочие роли Runbook, поддерживаемые виртуальными машинами Azure, включите стандартную защиту "отказ в обслуживании" (от атак DDoS) в виртуальных сетях, где размещены гибридные рабочие роли Runbook, чтобы защититься от атак от атак DDoS. С помощью интегрированной аналитики угроз центра безопасности Azure можно запретить обмен данными с известными вредоносными IP-адресами.  Настройте брандмауэр Azure для каждого сегмента виртуальной сети с включенной аналитикой угроз и настройте для **предупреждения и запрета** для вредоносного сетевого трафика.

Вы можете использовать безопасный сетевой доступ центра безопасности Azure, чтобы ограничить раскрытие виртуальных машин Windows разрешенными IP-адресами в течение ограниченного периода времени.  Кроме того, используйте адаптивные рекомендации по усилению защиты сети в центре безопасности Azure для конфигураций NSG, чтобы ограничить порты и исходные IP-адреса, исходя из фактического трафика и аналитики угроз.
